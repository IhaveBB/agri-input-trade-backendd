# 农资分类模块设计

## 一、设计目标

　　为满足农业生产资料在精细化管理中的实际需求，系统采用多级分类结构对农资商品进行管理。分类体系需满足：

1. 支持三级及以上扩展结构
2. 符合农业行业常见分类逻辑
3. 支持标准化管理与灵活扩展并存
4. 保证数据一致性与结构稳定性

　　系统采用树形层级结构设计。

## 二、分类层级结构设计

### （一）一级分类（宏观类别）

　　一级分类用于对农资进行行业层级划分，包括：

- 种子
- 农药
- 肥料
- 饲料
- 兽药
- 农膜
- 农机

　　一级分类为最高层级，parent\_id 为 NULL。

### （二）二级分类（功能类别）

　　二级分类隶属于一级分类，用于细化用途或属性。

　　例如：

　　种子：

- 粮食种植
- 蔬菜种子

　　农药：

- 杀虫剂
- 植物生长调节剂

### （三）三级分类（作物或细分品类）

　　部分二级分类支持三级细分。

　　例如：

　　种子 → 粮食种植：

- 谷物类
- 杂粮类
- 豆类作物

　　种子 → 蔬菜种子：

- 叶菜类
- 根茎类

## 三、数据结构设计

　　系统采用通用树形结构存储分类数据。

　　分类表设计如下：

- id
- name
- parent\_id
- level
- sort\_order
- status
- is\_custom（是否为商家自定义）
- create\_user\_id
- create\_time
- update\_time

　　说明：

1. level 字段标识当前层级（1、2、3…）
2. parent\_id 构建树形关系
3. is\_custom 用于区分系统预置分类与商家自定义分类

## 四、商家自定义分类设计说明

　　农业世界很复杂。某些地方性作物、特色品种可能未包含在系统预设分类中，因此允许商家申请新增一级分类以下的分类

　　但不能完全开放，否则分类会变成“文字垃圾场”。

　　规则如下：

1. 自定义分类默认状态为“待审核”
2. 管理员审核通过后转为正式分类
3. 存在同名分类不允许申请，新分类名称包含旧名称也不可，如存在小麦分类，那么优质小麦、高产小麦等无法申请
4. 自定义分类必须绑定父级分类
